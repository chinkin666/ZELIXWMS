# 05 开发计划 / 開発計画

> 3PL + FBA 通过型业务系统增强 / 3PL + FBA 通過型業務システム強化
> 综合 01-11 全部设计文档的最终开发计划
> 01-11 全設計ドキュメントを統合した最終開発計画

## 1. 核心原则 / コア原則

1. **先跑通 FBA 通过型一条线**，从客户上传到出货，再扩展 RSL/B2B/保管型
2. **基础先行**：客户模型 + 商品模型 + 权限模型先改完，再开发业务功能
3. **三端同步推进**：admin/ + portal/ + frontend/ 配合后端同步开发
4. **一个真实客户验证**：Phase 1 结束时用一个真实客户跑完整流程

## 2. Phase 总览 / Phase 概要

```
Phase 0 (基础)    ─── 10-12 天 ─── 模型改造 + 三端骨架 + 权限 + 照片存储
Phase 1 (通过型)  ─── 18-22 天 ─── FBA 通过型完整链路（门户→API→仓库）
Phase 2 (增强)    ─── 12-15 天 ─── 检品/贴标/异常/箱管理 + 模板设计器
Phase 3 (扩展)    ─── 10-12 天 ─── RSL/B2B + 保管型出库 + 盘点 + 库龄
Phase 4 (管理)    ─── 10-14 天 ─── KPI + 报告 + 大促 + 审计 + 引导

总计: 60-75 天
```

## 10. 风险与应对 / リスクと対策

| 风险 | 影响 | 应对 |
|------|------|------|
| 客户模型改造影响现有数据 | 现有订单/库存关联断裂 | Phase 0 写完善的迁移脚本 + 回归测试 |
| 三端同时开发资源不足 | 进度拖延 | admin/ 先只做客户管理+价格管理，其他功能 Phase 2+ 再加 |
| PDF 拆分在某些 PDF 上失败 | FBA 标打不出来 | 降级方案：客户选格式+固定坐标切，实在不行客户自己切成单张上传 |
| 热敏打印机适配困难 | 现场无法打标签 | 先支持 PDF 导出手动打印，后续接 ESC/POS |
| 第一个真实客户的需求超出设计 | 需要返工 | M2 之前预留 2-3 天 buffer 做调整 |
| 移动端适配工作量超预期 | 仓库现场不好用 | 仓库端先做 PC 版，移动端 Phase 1.5 补 |
