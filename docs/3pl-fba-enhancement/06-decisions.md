## ADR-001: FbaBox 使用独立 collection 而非嵌套数组

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
FBA 纳品计划需要管理箱级信息。有两种方案：在 FbaShipmentPlan 中用 nested array，或创建独立的 FbaBox collection。

**决定 / 決定**:
使用独立 collection。

**理由 / 理由**:
- 单个 plan 可能有 50-100 个箱，嵌套数组会导致 document 过大
- 分箱/合箱/换箱是高频操作，独立 collection 避免整个 plan document 的并发写入冲突
- 箱级操作（封箱、拍照、贴标）需要独立的状态机
- MongoDB 对 nested array 的原子更新支持有限（$elemMatch 只能匹配第一个）

**权衡 / トレードオフ**:
- 查询时需要额外的 populate/join
- 需要维护 fbaShipmentPlanId 外键一致性

## ADR-002: 照片存储使用 S3 兼容方案（MinIO → S3）

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
检品、异常、出货等环节需要大量照片存储。

**决定 / 決定**:
开发环境用 MinIO（Docker），生产环境用 AWS S3 Tokyo。使用 S3 兼容 API，一套代码两个环境。

**理由 / 理由**:
- MinIO 本地部署零成本，开发体验好
- S3 API 是事实标准，切换无代码改动
- 照片不存 MongoDB（避免 16MB document 限制，避免数据库膨胀）
- Tokyo region 延迟可控

**替代方案 / 代替案**:
- 本地文件系统：不可扩展，无 CDN，备份困难 → 否决
- MongoDB GridFS：增加数据库负担，查询混合 → 否决
- Cloudflare R2：更便宜但日本节点有限 → 备选

## ADR-003: 不直接对接 Amazon SP-API

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
可以通过 Amazon SP-API 自动获取纳品计划、FBA 库存、移除订单等数据。

**决定 / 決定**:
当前不对接 SP-API，由人工在 Seller Central 操作后手动录入系统。

**理由 / 理由**:
- SP-API 接入需要 Amazon Developer 审批，流程长
- 每个客户需要单独授权（OAuth），运营复杂度高
- 当前客户量小（<20），人工录入成本可接受
- 纳品计划数据量不大（每客户每周 1-5 个 plan）
- 优先把仓库内部流程做扎实，再考虑外部集成

**何时重新评估 / 再評価タイミング**:
- 客户数超过 50，或单客户月均 plan 超过 20 个
- 客户要求自动化对接

## ADR-004: 异常通知使用 webhook + 重试而非消息队列

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
SOP 要求异常必须及时通知客户，需要可靠投递。

**决定 / 決定**:
扩展现有 webhook 系统，增加重试机制（3 次指数退避）和送达日志。不引入 RabbitMQ/Redis Queue。

**理由 / 理由**:
- 系统已有 webhook 模块，扩展成本低
- 通知量不大（日均 <100 条），不需要消息队列
- 减少运维复杂度（不增加基础设施依赖）
- 3 次重试 + 人工升级已足够覆盖 SLA 要求

**权衡 / トレードオフ**:
- 服务重启时内存中的重试任务会丢失
- 应对: 定时任务每 15 分钟扫描未送达的通知补发

## ADR-005: 检品记录独立于入库单

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
检品信息可以作为 InboundOrderLine 的扩展字段，也可以独立为 InspectionRecord。

**决定 / 決定**:
创建独立的 InspectionRecord，通过 inboundOrderId + lineNumber 关联。

**理由 / 理由**:
- 检品有自己的生命周期（创建→执行→复核），不同于入库单
- 检品记录需要存储照片数组、多维度结果、异常列表，数据量大
- 同一入库行可能多次检品（首检 + 复检）
- 独立模型便于统计分析（检品合格率、异常趋势等）
- 未来可能扩展到出库检品、退货检品，模型可复用

## ADR-006: 移动端使用响应式 Web 而非原生 App

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
仓库现场需要移动端操作（扫码、拍照、提交）。

**决定 / 決定**:
使用响应式 Web 设计，核心操作页面适配移动端浏览器。不开发原生 App。

**理由 / 理由**:
- 开发成本低（一套代码）
- 部署简单（无需 App Store 审核）
- PDA 设备（如 Zebra、Keyence）通常内置浏览器
- 现代浏览器支持摄像头 API、扫码 API
- 已有 scanBeep 音频反馈基础

**限制 / 制限**:
- 离线能力有限（后续可考虑 PWA Service Worker）
- 扫码性能可能不如原生（可通过外接蓝牙扫码枪补偿）

**何时重新评估 / 再評価タイミング**:
- 离线需求成为刚需（WiFi 覆盖确实无法保障）
- 扫码性能成为瓶颈

## ADR-007: 租户隔离先在新模块实施，后推广

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
当前 tenantId 过滤靠各 controller 手动添加，有遗漏风险。

**决定 / 決定**:
新模块全部使用 Mongoose pre-query hook 自动注入 tenantId。老模块暂不改造，在 Phase 4 统一迁移。

**理由 / 理由**:
- 全局改造影响面大，可能引入回归 bug
- 新模块从一开始就正确实施，成本最低
- 老模块已运行稳定，短期风险可控
- Phase 4 有专门时间做统一迁移 + 回归测试

## ADR-008: 循环盘点采用随机选取而非 ABC 分类

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
循环盘点需要决定每月盘哪些 SKU。

**决定 / 決定**:
初始版本使用随机选取 20%（排除上月已盘），后续增加 ABC 分类优先级。

**理由 / 理由**:
- SOP 要求"每月 20%，季度全覆盖"，随机选取即可满足
- ABC 分类需要出库频率数据积累，当前数据不足
- 随机选取实现简单，不会有系统性遗漏
- 后续数据积累后，可切换为 A 类每月、B 类每季、C 类每半年

## ADR-009: 客户门户独立前端应用

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
客户门户与仓库管理端的用户群体、权限模型、页面逻辑完全不同。

**决定 / 決定**:
新建 `portal/` 独立前端项目，与 `frontend/`（仓库端）分离。共用 `backend/` API。

**理由 / 理由**:
- 用户群体不同：客户（中国卖家/日本公司）vs 仓库员工
- 权限模型不同：客户只看自己数据，仓库看全部
- 语言需求不同：客户端需要中/日/英三语，仓库端现有中日双语够用
- 部署独立：portal.zelix.com vs app.zelix.com，互不影响
- 避免单一 SPA 越来越臃肿

**权衡 / トレードオフ**:
- 两个前端项目有重复代码（UI 组件、API 调用）
- 后期可提取 shared/ 组件库解决

## ADR-010: 客户模型改造优先于功能开发

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
通过型流程、计费、门户都依赖 Client→SubClient→Shop 层级结构。

**决定 / 決定**:
Phase 1 第一步改造客户模型（新建 SubClient、Shop、PriceCatalog），然后再做检品、贴标等功能。

**理由 / 理由**:
- 所有入库预定都要挂 clientId + subClientId + shopId
- 所有费用明细都要按层级拆分
- 如果先做功能后改模型，已有数据全部要迁移补关联
- 客户模型改动量不大（2-3 天），但影响面最广

## ADR-011: PDF 拆分让客户选择格式，不做智能识别

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
Amazon FBA 外箱标 PDF 有 4-up 和 6-up 两种布局。

**决定 / 決定**:
客户上传时通过下拉框选择"4-up"/"6-up"/"单张"，系统按固定坐标切割。不做智能布局识别。

**理由 / 理由**:
- Amazon 的 PDF 格式标准，坐标固定，按网格切即可
- 智能识别需要 PDF 内容分析，投入产出比低
- 客户知道自己下载的是什么格式
- 如果识别错误导致标签贴错，后果严重

## ADR-012: 价格目录支持模板复制

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
每个客户独立定价，但新客户从零设定几十个价格项效率低。

**决定 / 決定**:
系统预设几套价格模板（通过型标准价、保管型标准价等），创建客户时选模板一键复制，再微调个别项。

**理由 / 理由**:
- 大部分客户价格差异不大，只有少数项需要调整
- 减少仓库管理员操作时间
- 避免漏设价格项导致免费作业

## ADR-013: 第一个客户只做 FBA 通过型，跑通再扩

**状态 / ステータス**: 已决定 / 決定済

**背景 / 背景**:
系统需要支持 FBA、RSL、B2B 三种目的地 + 通过型/保管型两种模式。

**决定 / 決定**:
优先只做 FBA 通过型一条线，从客户上传到出货完整跑通。RSL、B2B、保管型后续迭代。

**理由 / 理由**:
- FBA 是最核心也是量最大的场景
- 一条线跑通可以验证整个架构（客户门户→API→仓库端）
- 跑通后扩展到 RSL/B2B 只是换出货目的地，核心流程复用
- 避免同时开发多条线导致哪条都跑不通
