## 2. PostgreSQL 备份 / PostgreSQL バックアップ

### 2.1 Supabase 自动备份 / Supabase 自動バックアップ

Supabase Pro 计划提供每日自动备份。
Supabase Pro プランは日次自動バックアップを提供する。

- **频率 / 頻度**: 每日 1 次（UTC 00:00 左右）/ 毎日 1 回（UTC 00:00 頃）
- **保留 / 保持**: 7 天 / 7 日
- **恢复方式 / 復元方法**: Supabase Dashboard → Backups → Restore
- **注意 / 注意**: 恢复会覆盖当前数据库 / 復元は現在のデータベースを上書きする

### 2.2 手动 pg_dump / 手動 pg_dump

用于发布前或重大变更前的按需备份。
リリース前や重大変更前のオンデマンドバックアップに使用。

```bash
# 完整备份 / フルバックアップ
pg_dump "postgresql://postgres:[PASSWORD]@db.[PROJECT].supabase.co:5432/postgres" \
  --format=custom \
  --no-owner \
  --no-acl \
  -f backup_$(date +%Y%m%d_%H%M%S).dump

# 仅结构 / スキーマのみ
pg_dump "postgresql://..." --schema-only -f schema_$(date +%Y%m%d).sql

# 仅数据 / データのみ
pg_dump "postgresql://..." --data-only -f data_$(date +%Y%m%d).dump
```

**恢复 / 復元**:
```bash
pg_restore -d "postgresql://..." --clean --if-exists backup_20260321.dump
```

### 2.3 PITR（时间点恢复）/ PITR（ポイントインタイムリカバリ）

Supabase Pro 计划支持通过 WAL 进行时间点恢复。
Supabase Pro プランは WAL によるポイントインタイムリカバリをサポート。

- **RPO**: 数秒以内 / 数秒以内
- **恢复精度 / 復元精度**: 可恢复到任意时间点 / 任意の時点に復元可能
- **操作方法 / 操作方法**: Supabase Dashboard → Backups → Point in Time Recovery
- **保留期间 / 保持期間**: 最近 7 天 / 直近 7 日

## 3. Redis 备份 / Redis バックアップ

### 3.1 RDB 快照 / RDB スナップショット

```conf
# redis.conf
save 3600 1      # 每小时至少1次写入则保存 / 1時間で最低1回の書き込みで保存
save 300 100     # 5分钟内100次写入 / 5分以内に100回の書き込み
save 60 10000    # 1分钟内10000次写入 / 1分以内に10000回の書き込み
dbfilename dump.rdb
dir /data/redis/
```

### 3.2 AOF（追加写入文件）/ AOF（追記専用ファイル）

```conf
appendonly yes
appendfsync everysec    # 每秒同步，平衡性能与安全 / 毎秒同期、性能と安全のバランス
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

### 3.3 备份脚本 / バックアップスクリプト

```bash
#!/bin/bash
# Redis 定期备份 / Redis 定期バックアップ
BACKUP_DIR="/backups/redis"
DATE=$(date +%Y%m%d_%H%M%S)

redis-cli BGSAVE
sleep 5
cp /data/redis/dump.rdb "${BACKUP_DIR}/dump_${DATE}.rdb"

# 保留7天 / 7日間保持
find ${BACKUP_DIR} -name "dump_*.rdb" -mtime +7 -delete
```

## 4. 文件存储备份 / ファイルストレージバックアップ

Supabase Storage 基于 S3 兼容存储。
Supabase Storage は S3 互換ストレージに基づく。

- **版本管理 / バージョニング**: Supabase Storage はオブジェクトバージョニングをサポート
- **冗余 / 冗長性**: S3 标准冗余（99.999999999% 持久性）/ S3 標準冗長性（99.999999999% 耐久性）
- **跨区域复制 / クロスリージョンレプリケーション**: 需要时可通过 AWS S3 配置 / 必要に応じて AWS S3 で設定可能

```bash
# 手动下载备份 / 手動ダウンロードバックアップ
# 使用 Supabase CLI / Supabase CLI を使用
supabase storage ls --project-ref [PROJECT_REF]
```

## 5. 灾难恢复计划 / 災害復旧計画

### 5.1 恢复目标 / 復旧目標

| 指标 / 指標 | 目标 / 目標 | 说明 / 説明 |
|---|---|---|
| **RTO** | 4 小时 / 4 時間 | 从灾难发生到服务恢复 / 災害発生からサービス復旧まで |
| **RPO** | 1 小时 / 1 時間 | 最大可容忍数据丢失时间 / 最大許容データ損失時間 |

### 5.2 灾难等级 / 災害レベル

| 等级 / レベル | 描述 / 説明 | 示例 / 例 |
|---|---|---|
| L1 | 单一服务故障 / 単一サービス障害 | API pod 崩溃 / API pod クラッシュ |
| L2 | 多服务故障 / 複数サービス障害 | DB + API 同时故障 / DB + API 同時障害 |
| L3 | 完全宕机 / 完全停止 | 数据中心级别故障 / データセンターレベル障害 |

### 5.3 恢复步骤 / 復旧手順

#### L1: 单一服务故障 / 単一サービス障害

1. 确认故障服务 / 障害サービスを確認
2. 检查日志（Pino 结构化日志）/ ログ確認（Pino 構造化ログ）
3. 尝试重启服务 / サービス再起動を試行
4. 如果重启无效，回滚到上一版本 / 再起動が無効なら前バージョンにロールバック
5. 验证服务健康检查 / ヘルスチェックで確認

#### L2: 多服务故障 / 複数サービス障害

1. 启动事件响应流程 / インシデント対応プロセスを開始
2. 确认受影响范围 / 影響範囲を確認
3. 检查 Supabase 状态页面 / Supabase ステータスページを確認
4. 从备份恢复数据库（如需要）/ 必要に応じてバックアップからDB復元
5. 按依赖顺序重启服务 / 依存順序でサービスを再起動
   - PostgreSQL → Redis → Backend API → Frontend
6. 运行 E2E 健康检查 / E2E ヘルスチェックを実行
7. 确认所有租户数据完整 / 全テナントデータの完全性を確認

#### L3: 完全宕机 / 完全停止

1. 启动灾难恢复通信计划 / DR 通信計画を開始
2. 通知所有利益相关者 / 全関係者に通知
3. 使用备用 Supabase 项目（如已预配置）/ 予備 Supabase プロジェクトを使用（事前設定済みの場合）
4. 从最近的 pg_dump 恢复 / 最新の pg_dump から復元
5. 恢复 Redis 数据 / Redis データを復元
6. 重新部署所有服务 / 全サービスを再デプロイ
7. 数据完整性检查 / データ整合性チェック
8. 逐步恢复流量 / 段階的にトラフィックを復旧

### 5.4 通信计划 / 通信計画

| 时间点 / タイミング | 动作 / アクション | 对象 / 対象 |
|---|---|---|
| 发生后 5 分钟 / 発生後 5 分 | 确认事件并通知团队 / インシデント確認しチームに通知 | 开发团队 / 開発チーム |
| 发生后 15 分钟 / 発生後 15 分 | 初步影响评估 / 初期影響評価 | 管理层 / マネジメント |
| 发生后 30 分钟 / 発生後 30 分 | 客户通知（状态页更新）/ 顧客通知（ステータスページ更新） | 客户 / 顧客 |
| 每小时 / 毎時 | 进展更新 / 進捗更新 | 全部 / 全員 |
| 恢复后 / 復旧後 | 事后报告（RCA）/ ポストモーテム（RCA） | 全部 / 全員 |

## 6. DR 演练 / DR 演習

**频率 / 頻度**: 每季度 1 次 / 四半期に 1 回

### 演练流程 / 演習フロー

1. **计划 / 計画**（演练前 1 周 / 演習 1 週間前）
   - 确定演练场景 / 演習シナリオを決定
   - 通知相关人员 / 関係者に通知
   - 准备演练环境 / 演習環境を準備

2. **执行 / 実行**
   - 模拟故障场景 / 障害シナリオをシミュレーション
   - 按照恢复步骤操作 / 復旧手順に従い操作
   - 记录实际 RTO/RPO / 実際の RTO/RPO を記録

3. **评估 / 評価**
   - 对比目标 RTO/RPO 与实际值 / 目標と実績の RTO/RPO を比較
   - 识别改进点 / 改善点を特定
   - 更新文档 / ドキュメントを更新

### 演练检查项 / 演習チェック項目

- [ ] pg_dump 备份可正常恢复 / pg_dump バックアップが正常に復元可能
- [ ] PITR 恢复到指定时间点 / PITR が指定時点に復元可能
- [ ] Redis RDB 恢复正常 / Redis RDB 復元が正常
- [ ] 全服务在 RTO 内恢复 / 全サービスが RTO 内に復旧
- [ ] 数据完整性检查通过 / データ整合性チェックが通過
- [ ] 通信计划执行正确 / 通信計画が正しく実行
