# 变更管理 / 変更管理

> 系统变更的分类、审批与执行流程
> システム変更の分類、承認、実行フロー

---

## 1. 变更分类 / 変更分類

| 分类 / 分類 | 审批要求 / 承認要件 | 示例 / 例 |
|---|---|---|
| **标准变更 / 標準変更** | 无需审批（预批准）/ 承認不要（事前承認済み） | 依赖更新、文档修改、配置微调 / 依存関係更新、ドキュメント修正、設定微調整 |
| **常规变更 / 通常変更** | 同伴审查（PR review）/ ピアレビュー（PR review） | 新功能、Bug修复、重构 / 新機能、バグ修正、リファクタリング |
| **紧急变更 / 緊急変更** | 事后审查 / 事後レビュー | 生产环境紧急修复 / 本番環境緊急修正 |

---

## 2. 标准变更（预批准）/ 標準変更（事前承認済み）

以下变更类型已预先批准，无需额外审批流程。
以下の変更タイプは事前承認済み、追加承認プロセス不要。

- npm 依赖的 patch/minor 更新（无破坏性变更）/ npm 依存関係の patch/minor 更新（破壊的変更なし）
- 文档更新 / ドキュメント更新
- 日志级别调整 / ログレベル調整
- 环境变量的非敏感值更新 / 環境変数の非機密値更新
- 测试用例添加 / テストケース追加

**要求 / 要件**: 仍需遵循 Git 工作流，提交 PR。
Git ワークフローに従い、PR を提出すること。

---

## 3. 常规变更流程 / 通常変更フロー

### Step 1: 创建变更请求 / 変更リクエストを作成

在 GitHub Issues 中创建 Issue，包含以下信息。
GitHub Issues で Issue を作成し、以下の情報を含める。

```markdown
## 变更描述 / 変更説明
（变更的内容和目的 / 変更の内容と目的）

## 影响评估 / 影響評価
- 影响的模块 / 影響を受けるモジュール:
- 影响的租户 / 影響を受けるテナント: 全部 / 特定
- 是否涉及 DB 变更 / DB 変更の有無: 是/否
- 是否涉及 API 变更 / API 変更の有無: 是/否

## 回滚计划 / ロールバック計画
（如何回滚此变更 / この変更のロールバック方法）

## 测试计划 / テスト計画
（如何验证变更正确性 / 変更の正確性をどう検証するか）
```

### Step 2: 影响评估 / 影響評価

| 评估项 / 評価項目 | 确认内容 / 確認内容 |
|---|---|
| 模块影响 / モジュール影響 | 列出受影响的模块 / 影響を受けるモジュールを列挙 |
| 租户影响 / テナント影響 | 是否影响所有租户 / 全テナントに影響するか |
| 数据影响 / データ影響 | 是否涉及数据迁移或结构变更 / データ移行やスキーマ変更があるか |
| 性能影响 / パフォーマンス影響 | 是否可能影响性能 / パフォーマンスに影響する可能性があるか |
| 安全影响 / セキュリティ影響 | 是否涉及认证、授权、数据访问 / 認証、認可、データアクセスに関わるか |

### Step 3: 开发与审查 / 開発とレビュー

```bash
# 遵循标准 Git 工作流 / 標準 Git ワークフローに従う
git checkout -b feat/change-description
# ... 开发 / 開発 ...
git push -u origin feat/change-description
gh pr create
```

PR 必须获得至少 1 位审查者的批准。
PR は最低 1 名のレビュアーの承認が必要。

### Step 4: Staging 验证 / Staging 検証

合并到 develop 后在 staging 环境验证。
develop にマージ後、staging 環境で検証。

- [ ] 功能正常工作 / 機能が正常に動作
- [ ] 无回归问题 / リグレッションなし
- [ ] 性能无明显下降 / パフォーマンスの著しい低下なし

### Step 5: 生产部署 / 本番デプロイ

在维护窗口内部署到生产环境。
メンテナンスウィンドウ内に本番環境にデプロイ。

---

## 4. 紧急变更流程 / 緊急変更フロー

用于生产环境的紧急修复（P1/P2 事件）。
本番環境の緊急修正に使用（P1/P2 インシデント）。

1. **立即修复 / 即座に修正** — 创建 hotfix 分支，最小化修改 / hotfix ブランチを作成、最小限の変更
2. **快速测试 / 迅速テスト** — 运行核心测试套件 / コアテストスイートを実行
3. **直接部署 / 直接デプロイ** — 跳过 staging，直接部署到生产 / staging をスキップ、直接本番にデプロイ
4. **事后审查 / 事後レビュー** — 24 小时内提交 PR 进行审查 / 24 時間以内に PR を提出しレビュー
5. **RCA 报告 / RCA レポート** — 48 小时内完成根因分析 / 48 時間以内に根本原因分析を完了

## 5. 维护窗口 / メンテナンスウィンドウ

| 项目 / 項目 | 值 / 値 |
|---|---|
| **时间 / 時間** | 每周二、周四 02:00-04:00 JST / 毎週火曜・木曜 02:00-04:00 JST |
| **通知 / 通知** | 至少 24 小时前通知 / 最低 24 時間前に通知 |
| **范围 / 範囲** | 数据库迁移、基础设施变更、大型部署 / DB マイグレーション、インフラ変更、大規模デプロイ |

**维护窗口外的部署 / メンテナンスウィンドウ外のデプロイ**:
- 标准变更和无 DB 变更的常规变更可在窗口外部署 / 標準変更と DB 変更なしの通常変更はウィンドウ外にデプロイ可
- 紧急变更不受窗口限制 / 緊急変更はウィンドウに制限されない

## 6. 数据库 Schema 变更 / データベーススキーマ変更

**原则: 始终向后兼容 / 原則: 常に後方互換**

采用 Expand-then-Contract 模式。
Expand-then-Contract パターンを採用。

### 添加列 / カラム追加

```sql
-- Phase 1: Expand（添加可为空的新列）/ Expand（NULL 許容の新カラムを追加）
ALTER TABLE products ADD COLUMN new_field VARCHAR(255);

-- Phase 2: 代码更新（同时写入新旧列）/ コード更新（新旧両カラムに書き込み）
-- 部署新代码 / 新コードをデプロイ

-- Phase 3: 数据迁移（填充新列）/ データ移行（新カラムにデータを埋める）
UPDATE products SET new_field = old_field WHERE new_field IS NULL;

-- Phase 4: Contract（移除旧列，仅在确认安全后）/ Contract（旧カラムを削除、安全確認後のみ）
-- 至少等待 1 个发布周期 / 最低 1 リリースサイクル待つ
ALTER TABLE products DROP COLUMN old_field;
```

### 禁止操作 / 禁止操作

- **禁止 / 禁止**: 直接重命名列 / カラムの直接リネーム
- **禁止 / 禁止**: 直接删除有数据的列 / データのあるカラムの直接削除
- **禁止 / 禁止**: 更改列类型为不兼容类型 / 非互換型へのカラム型変更
- **禁止 / 禁止**: 添加带默认值的 NOT NULL 列到大表 / 大テーブルへの DEFAULT 付き NOT NULL カラム追加

## 7. Feature Flags / フィーチャーフラグ

对于高风险或大型功能变更，使用 Feature Flags 进行渐进式发布。
高リスクまたは大規模な機能変更には、Feature Flags で段階的リリースを行う。

### 发布策略 / リリース戦略

```
Phase 1: 内部测试 (团队成员) / 内部テスト（チームメンバー）
  ↓
Phase 2: Beta (选定租户, 10%) / Beta（選定テナント, 10%）
  ↓
Phase 3: 渐进发布 (50%) / 段階的リリース（50%）
  ↓
Phase 4: 全量发布 (100%) / 全量リリース（100%）
  ↓
Phase 5: 清理 Feature Flag / Feature Flag のクリーンアップ
```

### 清理规则 / クリーンアップルール

- Feature Flag 全量发布后 **2 个发布周期内**必须清理代码中的 Flag 判断
  Feature Flag 全量リリース後 **2 リリースサイクル以内**にコード内の Flag 判定を除去必須
- 未清理的 Feature Flag 视为技术债务 / 未清理の Feature Flag は技術的負債とみなす
