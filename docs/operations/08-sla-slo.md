|---|---|
| **月度可用性 / 月次可用性** | 99.9% |
| **允许停机时间 / 許容ダウンタイム** | 43.8 分钟/月 / 43.8 分/月 |

| **适用范围 / 適用範囲** | 所有 API 端点和 Web UI / 全 API エンドポイントと Web UI |

## 3. SLI（服务级别指标 / サービスレベル指標）

### 3.1 测量方法 / 測定方法

| SLO | SLI 测量 / SLI 測定 | 数据来源 / データソース |
|---|---|---|
| API 可用性 | 成功请求数 / 总请求数 / 成功リクエスト数 / 総リクエスト数 | 健康检查 + API 日志 / ヘルスチェック + API ログ |
| API 延迟 / レイテンシ | 请求处理时间百分位 / リクエスト処理時間パーセンタイル | Pino 日志中的 responseTime |
| 错误率 / エラー率 | 5xx 响应数 / 总响应数 / 5xx レスポンス数 / 総レスポンス数 | API 日志 |
| 数据持久性 / データ耐久性 | 备份成功率 + 恢复测试 / バックアップ成功率 + 復元テスト | Supabase + DR 演练 |

### 3.2 可用性计算 / 可用性計算

```
月度可用性 = (总分钟数 - 停机分钟数) / 总分钟数 × 100%
月次可用性 = (総分数 - ダウンタイム分数) / 総分数 × 100%

例: 30天の月 / 30日の月
总分钟 / 総分数 = 30 × 24 × 60 = 43,200
99.9% → 允许停机 / 許容ダウンタイム = 43.2 分
99.95% → 允许停机 / 許容ダウンタイム = 21.6 分
```

## 4. 错误预算 / エラーバジェット

### 4.1 定义 / 定義

错误预算 = 100% - SLO 目标。在预算内允许承担风险（部署、实验等）。
エラーバジェット = 100% - SLO 目標。バジェット内でリスクを取ることが許容される（デプロイ、実験等）。

| SLO | 错误预算 / エラーバジェット | 每月允许 / 月次許容 |
|---|---|---|
| 99.9% 可用性 | 0.1% | 43.2 分钟停机 / 43.2 分ダウンタイム |
| 0.1% 错误率 / エラー率 | 0.1% | 每 1000 请求 1 个错误 / 1000 リクエストに 1 エラー |

### 4.2 错误预算策略 / エラーバジェットポリシー

**预算充足时（> 50% 剩余）/ バジェット余裕あり（> 50% 残り）**:
- 正常发布节奏 / 通常のリリース頻度
- 允许实验性功能 / 実験的機能を許可
- 可以进行基础设施变更 / インフラ変更が可能

**预算紧张时（< 30% 剩余）/ バジェット逼迫（< 30% 残り）**:
- 减少发布频率 / リリース頻度を削減
- 优先修复可靠性问题 / 信頼性問題の修正を優先
- 增加测试覆盖 / テストカバレッジを増加

**预算耗尽时（0% 剩余）/ バジェット枯渇（0% 残り）**:
- 冻结非关键发布 / 非クリティカルリリースを凍結
- 全力投入可靠性改善 / 信頼性改善に全力投入
- 进行事后分析 / ポストモーテムを実施
